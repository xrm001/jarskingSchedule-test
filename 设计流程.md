# 石总行程与预约应用：任务设计流程

## 1. 录音结论

这不是单纯的“会议室预约”，而是一个以老板行程为核心的内部预约平台：同事先看可用时间，再提交预约；老板审批后占用时间；所有人能看到可用性，但私密会议只暴露“忙碌”；系统在每日早上和会前主动提醒老板。

## 2. 角色与权限

| 角色 | 能做什么 |
| --- | --- |
| 管理层 | 查看空闲与占用、提交预约、查看自己的申请、取消未开始的预约 |
| 石总 | 自建行程、切换状态、语音录入、审批/拒绝、查看每日摘要 |
| 管理员 | 维护人员权限和会议室、查看审计记录；不能代替石总审批 |

可见范围分三级：管理层可见、仅显示占用、仅老板和管理员可见。获批会议或个人行程都必须占用时间。

## 3. 主业务流程

1. 老板或秘书录入不可预约的个人行程。
2. 同事打开公共日程，选择日期并查看空闲时间。
3. 选择空闲时间，填写主题、地点、参与人、可见范围和备注。
4. 后端校验已通过日程和会议室；待审批申请不锁定时段，允许多人申请相同或重叠时间。
5. 企业微信应用消息通知石总，石总选择通过或拒绝；拒绝不要求填写理由。
6. 通过后，石总日程和会议室日程同时锁定；所有时间重叠的待审批申请自动拒绝并分别通知申请人。
7. 每天09:00给石总推送当日摘要；会议前60分钟及10分钟再次提醒。

状态机：草稿 → 待审批 → 已通过 / 已拒绝；已通过 → 已取消 / 已完成。超过开始时间仍未审批应自动过期。

## 4. 页面信息架构

- 日程首页：周日期、当日安排、空闲时段、状态与隐私标识。
- 新建预约：主题、日期时间、会议室、参与人、可见范围、提醒、备注。
- 审批中心：按重叠时段分组的待审批列表、详情、通过和直接拒绝。
- 我的预约：按状态筛选、取消、再次预约。
- 老板视角：今日摘要、自建行程、语音录入、状态切换、待审批数量、日/周视图。
- 管理后台：会议室资源、人员权限、提醒策略、操作日志。

## 5. 冲突与隐私规则

- 同一老板同一时间只能有一条有效日程；同一会议室同一时间只能被一条已通过预约占用。
- 待审批不占位；多人可同时申请相同或重叠时间。
- 审批使用数据库事务：通过一条后锁定资源，并自动拒绝全部重叠待审批申请，避免“双订”。
- 私密日程对无权限者只显示日期、时间和“已占用”，不显示主题、地点、参与人。
- 查看和修改私密日程均写入审计日志。

## 6. 通知方案

优先做企业微信自建应用/H5，而不是单独开发 iOS App：它能统一身份、组织架构、应用消息和移动端入口。消息节点包括提交成功、待老板审批、审批结果、变更/取消、每日摘要、会前提醒和勿扰超时提醒。个人微信通知能力受平台限制，不应作为首版主链路。

## 7. MVP 开发顺序

1. 企业微信登录、组织成员与角色权限。
2. 老板日历、空闲/占用展示、私密脱敏。
3. 新建预约、冲突校验、审批状态机。
4. 企业微信消息通知和每日/会前定时任务。
5. 我的预约、取消/改期、操作日志。
6. 第二阶段再做会议室设备、Bot 自然语言查询、日历同步、统计报表。

## 8. 开发前必须确认

- 预约对象只有石总，还是未来支持多个领导？
- 待审批时段是否临时锁定，锁定多久？
- 谁能看到“公开详情”，普通员工还是仅管理层？
- 老板拒绝是否必须填写理由？谁能代老板审批？
- 会议室是独立资源，还是只记录地点名称？
- 每日摘要时间和会前提醒时间是否允许个人配置？

## 9. 验收标准

- 被占用或冲突的时段不可提交。
- 私密行程在无权限账号下不泄露主题、参与人和地点。
- 审批结果能实时反映到公共日程，并通知申请人。
- 每日摘要及会前提醒仅发送一次且可追踪送达结果。
- 所有新增、修改、审批、取消操作都有操作者与时间记录。
